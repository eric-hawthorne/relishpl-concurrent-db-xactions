Imports

Your relish program can import code from elsewhere, in order to use the data types, methods, and constants of the other code.

relish software comes in units called artifacts, subdivided into packages.
An artifact can be a software library or framework, or an application.
A package is a single module in the artifact. A package is a set of code files in a single directory. A package is a namespace for the data types and constants that it defines. If a package's (path)name is game/physics/motion, then constants in the package are used as motion.MAX_SPEED in the importing program, and data types are used as motion.VelocityVector

You import one package at a time into your program, with a package import declaration. The import declaration simply specifies the name of the package. There are three cases determining how package names are formatted in the import declaration.

  • If the package is a relish standard-library package, then the import declaration is only the package name (package pathname within the standard-library artifact).
  • If the package is in the same artifact as your program, then the import declaration is only the package name (package pathname within its artifact).
  • If the package is in a different software artifact, then the import declaration must be the artifact's full name followed by '/pkg/' then the package pathname.

Example

import
   strings
   game/physics/motion 
   game/physics/geometry as geom   
   generous.opensource.developer.org2011/game_naive_physics/pkg/phys_obj/gravity

Note: In the simple program trial environment, leave out the 'import' line and don't indent the actual import declarations.

The first import is of the relish standard-library strings package. This makes String manipulating methods available to the program.

The second import line above shows an import of a package from the same software artifact as the current program. Note that imports like this are not possible in the trial program environment, because a trial program is a whole artifact to itself, with no packages other than your trial program in it.

The third import line above shows an import of a package from the same software artifact as the current program. This package, whose name inside this source file would normally be geometry, as in geometry.Polygon, is aliased so that it is known in this source file as geom, as in geom.Polygon.

The fourth import line above shows an import of a package from a different software artifact. The other software artifact's originating organization is called generous.opensource.developer.org2011 . The relish software artifact they developed is called game_naive_physics, presumably a code library of physics objects and calculation methods. The package we are importing is called phys_obj/gravity and it will be known just as gravity when referred to in this source code file. As in gravity.G
